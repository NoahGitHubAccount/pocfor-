# 智慧分案系統 POC 重建：完整知識庫

---

## 第一章：這個系統在做什麼

### 背景與問題

**情境：**
台東縣政府每天收到大量民眾陳情案件，需要人工判斷應分派給哪個局處。

**痛點：**
- 人工分案耗時、容易出錯
- 分類標準因人而異，主觀性高
- 案件量大時回應速度慢

**解決方案：**
用 AI 自動讀取陳情文字，預測應分派的局處，提供承辦人員參考建議。

---

### 系統功能與輸出

**流程：**
```
民眾陳情文字
      ↓
  AI 分析文字
      ↓
輸出預測局處（含信心值排名）
      ↓
  承辦人員確認分案
```

**分類目標（9 個局處）：**

| 編號 | 局處名稱 |
|------|----------|
| 0 | 臺東市公所 |
| 1 | 交通及觀光發展處 |
| 2 | 社會處 |
| 3 | 建設處 |
| 4 | 國際發展及計畫處 |
| 5 | 教育處 |
| 6 | 農業處 |
| 7 | 環境保護局 |
| 8 | 警察局 |

**輸出範例：**
```
輸入：「檢舉太麻里補習班體罰學生」
→ 第1名：教育處     信心值 0.51
→ 第2名：警察局     信心值 0.20
→ 第3名：社會處     信心值 0.18
```

## 第三章：核心 AI 技術概念

### 什麼是詞向量（Word Embedding）

💬 **白話：**
電腦不懂中文，只懂數字。詞向量就是把每個詞翻譯成一串數字，讓意思相近的詞，數字也相近。

```
「警察」→ [0.2,  0.8, -0.3, 0.1, ...]  ← 100 個數字
「警局」→ [0.2,  0.7, -0.3, 0.1, ...]  ← 跟「警察」很像
「農業」→ [-0.5, 0.1,  0.9, -0.2, ...]  ← 跟「警察」差很多
```

**100d 是什麼意思？**
每個詞用 100 個數字來描述，d = dimension（維度）。

**gensim word2vec：**
用來訓練詞向量的工具。餵入大量文字後，自動學出詞語間的語意關係。

**本專案詞向量的價值：**
舊系統已用台東縣案件資料訓練出專屬詞向量，認識「太麻里」「環保局」等台東地名與政府單位，是珍貴資產，POC 重建時優先複用。

---

### 什麼是 Epochs（訓練迭代次數）

💬 **白話：**
Epoch 就是「把所有訓練資料完整看過一遍」。

```
Epoch 1  = 把 23,329 筆案件從頭看到尾（第一遍）
Epoch 2  = 再看一遍（第二遍）
Epoch 3  = 再看第三遍...
  ↓
Epoch 100 = 看了 100 遍
```

每看完一遍，模型就調整一次參數，讓預測結果更準確。就像反覆讀書，讀得越多越熟練。

**但不是越多越好：**

| 情況 | 問題 | 比喻 |
|------|------|------|
| Epochs 太少 | 還沒學夠，準確度低（欠擬合） | 書只讀了一遍就去考試 |
| Epochs 太多 | 把訓練資料「背起來」了，遇到新資料反而不準（過擬合） | 死背題目，換題型就不會 |
| Epochs 剛好 | 準確度高，遇到新資料也能正確預測 | 真正理解，舉一反三 |

**與 batch_size 的關係：**

模型不是一次看完所有資料才調整，而是每看一批（batch）就調整一次：

```
batch_size = 64   ← 每次看 64 筆，算一次，再看下 64 筆

每個 epoch 的更新次數 = 23,329 ÷ 64 ≈ 364 次
總更新次數 = 364 × 100 epochs = 36,400 次調整
```

💬 **白話：** 不是讀完整本書才做筆記，而是每讀 64 頁就整理一次重點，這樣學習比較穩定有效率。

**本專案設定：**

| 參數 | 數值 | 說明 |
|------|------|------|
| epochs | 100 | 完整看過訓練資料 100 遍 |
| batch_size | 64 | 每批 64 筆資料更新一次 |

### 什麼是 TextCNN（方向 A）

💬 **白話：**
CNN 原本用來看圖片，這裡改成「看文字」。用不同大小的視窗滑過句子，找出重要的詞語組合。

**模型流程：**
```
輸入文字：「檢舉太麻里補習班體罰學生」
      ↓
  jieba 斷詞
      ↓
[「檢舉」,「太麻里」,「補習班」,「體罰」,「學生」]
      ↓
  每個詞換成 100 維詞向量
      ↓
  組成 600×100 的矩陣（句子最長 600 詞）
      ↓
  用 2字、3字、4字視窗滑過（卷積）
      ↓
  max pooling 取最重要特徵
      ↓
  Dropout 防止過擬合
      ↓
  Softmax 輸出 9 個局處的機率
```

**模型關鍵參數：**

| 參數 | 數值 | 說明 |
|------|------|------|
| embedding_size | 100 | 詞向量維度 |
| seq_length | 600 | 句子最大長度 |
| num_classes | 9 | 分類數量 |
| filter_sizes | [2,3,4] | 卷積核視窗大小 |
| num_filters | 128 | 卷積核數量 |
| keep_prob | 0.5 | Dropout 保留率 |
| batch_size | 64 | 每批訓練樣本數 |
| learning_rate | 1e-3 | 學習速率 |

**優點：** 快速、輕量、CPU 可跑
**限制：** 不理解上下文語意（「不違規」和「違規」可能被視為相似）

---

### 什麼是 BERT Fine-tune（方向 B）

💬 **白話：**
BERT 是 Google 訓練好的超級語言學家，讀過幾百億句中文。Fine-tune 就是讓這位語言學家去台東縣政府「實習幾週」，專門學習分案語境。

**Fine-tune 三步驟：**
```
步驟1：下載預訓練 bert-base-chinese
       （Google 訓練好的，認識所有中文語意）
      ↓
步驟2：加上分類層（對應 9 個局處）
      ↓
步驟3：用台東案件資料繼續訓練幾個 epoch
       （微調方向，不需從頭學）
```

**與 TextCNN 的關鍵差異：**

| 比較項目 | TextCNN | BERT Fine-tune |
|----------|---------|----------------|
| 語意理解 | 局部詞語組合 | 整句上下文 |
| 「不違規」vs「違規」 | 可能混淆 | 能正確區分 |
| 訓練速度 | 快 | 慢（無 GPU 更慢）|
| 資源需求 | CPU 可跑 | 建議有 GPU |
| 準確度 | 好 | 通常更好 |
| 起點 | 從零學起 | 已有深厚語言基礎 |
| 斷詞需求 | 需要 jieba | 不需要，BERT 自己處理 |

💬 **比喻：**
- TextCNN = 用放大鏡「局部掃描」文字找關鍵詞
- BERT = 整句話「完整讀懂再思考」

**為什麼先做 A 再做 B？**
TextCNN 讓你先理解完整流程（資料→訓練→預測→API），建立基準準確度。BERT 再來對比，才知道升級值不值得。

## 第四章：重建技術選型

### 技術選型決策表

| 項目 | 舊系統 | 重建目標 | 決策理由 |
|------|--------|----------|----------|
| 執行環境 | CentOS bare-metal | Docker container | 可攜、可重現，換機器不用重裝 |
| Python | 3.6.8 | 3.11 | 容器內指定，不影響本機環境 |
| 深度學習框架 | TensorFlow 1.14 | PyTorch 2.x | 社群活躍、寫法直覺、易 debug |
| 中文斷詞 | jieba 0.39 | jieba 最新版 | 相容，直接升級 |
| 詞向量 | gensim word2vec | 複用舊向量或重訓 | 視準確度結果決定 |
| Web API | Flask 1.1.2 | FastAPI + uvicorn | 自動文件、速度快、型態驗證 |
| 部署 | 手動安裝 | Dockerfile + docker-compose | 一個指令啟動全部服務 |

---

### TensorFlow vs PyTorch

💬 **白話：** 兩者都是「蓋神經網路的工具箱」，差別在思考方式。

| | TensorFlow 1.14（舊） | PyTorch 2.x（新） |
|---|---|---|
| 寫法風格 | 先畫完整設計圖才能動工 | 邊做邊調整，像一般 Python |
| Debug 難度 | 難，錯誤訊息不直覺 | 容易，隨時可以印出中間結果 |
| 社群現況 | 舊版幾乎無人維護 | 學術界、業界主流 |
| 安裝相容性 | 只支援 Python 3.6 | Python 3.11 正常安裝 |

---

### Flask vs FastAPI

💬 **白話：** 兩者都是讓 AI 能接受網路請求的「服務窗口」。

| | Flask 1.1.2（舊） | FastAPI + uvicorn（新） |
|---|---|---|
| API 文件 | 要自己手動寫 | 自動產生，開 /docs 就能看到 |
| 執行速度 | 一般，同步處理 | 快，非同步設計 |
| 型態驗證 | 無 | 自動擋掉格式錯誤的請求 |
| 學習曲線 | 簡單 | 稍複雜但現代 |

---

### Docker 概念

💬 **白話：** Docker 就像便當盒。把程式、套件、設定全部裝進同一個便當盒，不管在哪台電腦打開，裡面的東西都一樣。

**本專案的 Docker 策略：**
```
docker-compose.yml
  ├── train 服務  ← 一次性執行，跑完就停，產出模型到 checkpoints/
  └── api 服務    ← 常駐執行，載入模型，接受 /predict 請求
```

**訓練資料處理方式：** 透過 volume mount 掛載到容器，不把資料 COPY 進 image（資料不進版控）

**對外 Port：** 8080（避免與現有 n8n_poc 等服務衝突）
