## 1. 系統全貌

### 1.1 系統目的

接收市民陳情文字，自動預測應分派的局處，回傳 Top-N 候選結果與信心值。原人工初分流程由 AI 輔助，最終決定仍由人工確認。

### 1.2 資料流

```
使用者輸入陳情文字
        │
        ▼
  POST /predict
  {"preString": "...", "preNum": 3}
        │
        ▼
  Tokenizer / 前處理
  （BERT：AutoTokenizer | TextCNN：jieba + vocab）
        │
        ▼
  模型推論
  （BERT：AutoModelForSequenceClassification | TextCNN：CNN + Softmax）
        │
        ▼
  Top-N Softmax 機率
        │
        ▼
  回傳 JSON
  {"case": [{"id": 2, "ou": "建設處", "probability": "0.8889..."}]}
```

### 1.3 兩方案並存架構

本 POC 同時實作兩個獨立方案，可視需求擇一或並行部署：

| 項目 | TextCNN（`poc/`） | BERT（`poc-bert/`） |
|------|-----------------|-------------------|
| 模型類型 | 卷積神經網路 + Word2Vec | Transformer 預訓練模型微調 |
| 預訓練基礎 | 自訓 Word2Vec（8,000 詞） | `hfl/chinese-roberta-wwm-ext`（HuggingFace） |
| 斷詞方式 | jieba 手動斷詞 | BERT Tokenizer（自動，不需 jieba） |
| API Port | **8080** | **8081** |
| 模型大小 | ~4 MB | ~400 MB |
| 測試集準確度 | 62% | 76% |
| Macro F1 | 0.58 | 0.73 |
| 推論速度 | 極快（< 100ms） | < 1 秒/筆 |

**建議**：正式部署採用 BERT 方案（Port 8081），TextCNN 保留供比較與快速實驗。

## 2. 部署與維運

### 2.1 環境需求

| 項目 | 需求 |
|------|------|
| 作業系統 | Linux / WSL2（Rocky9 或 Ubuntu） |
| Docker | 20.10 以上 |
| Docker Compose | v2 以上 |
| GPU（推論） | 非必要，CPU 可跑（< 1 秒/筆） |
| GPU（訓練） | 建議 NVIDIA 8GB+ VRAM |

### 2.2 啟動服務

```bash
# BERT API（Port 8081，推薦）
cd poc-bert
docker compose up -d api

# TextCNN API（Port 8080）
cd poc
docker compose up -d api
```

兩個服務可同時運行，互不衝突。

### 2.3 確認服務狀態

```bash
# 健康檢查
curl http://localhost:8081/health
# 預期回傳：{"status":"ok","model_loaded":true,"model_type":"BERT (chinese-roberta-wwm-ext)"}

# 查看 container 狀態
docker ps | grep api

# 查看 log
docker logs bert-api
docker logs textcnn-api
```

### 2.4 停止服務

```bash
docker compose -f poc-bert/docker-compose.yml down
docker compose -f poc/docker-compose.yml down
```

### 2.5 常見異常排查

| 症狀 | 可能原因 | 處理方式 |
|------|---------|---------|
| `/predict` 回傳 503 | 模型未載入 | 確認 `checkpoints/` 目錄有模型檔、重啟 container |
| Container 啟動後立即退出 | 模型檔案路徑錯誤 | `docker logs <container>` 查看錯誤訊息 |
| 推論速度突然變慢 | 記憶體不足，swap 取代 GPU/RAM | `docker stats` 確認記憶體使用量 |
| BERT 模型載入失敗 | `bert-model/` 目錄不完整 | 確認目錄內有 `model.safetensors` 與 `config.json` |

> **詳細部署步驟指引**：待補（包含 GPU 機器訓練後搬回的完整流程）

## 3. API 規格

### 3.1 POST `/predict` — 陳情分類

**請求**

```json
{
  "preString": "路燈故障無法修復，已多次反映",
  "preNum": 3
}
```

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| preString | string | ✅ | 陳情文字 |
| preNum | integer | 否（預設 3） | 回傳幾個候選局處，範圍 1–9 |

**回應**

```json
{
  "case": [
    {"id": 2, "ou": "建設處",         "probability": "0.8889628053"},
    {"id": 6, "ou": "臺東市公所",      "probability": "0.0548732281"},
    {"id": 4, "ou": "環境保護局",      "probability": "0.0234150812"}
  ]
}
```

| 欄位 | 說明 |
|------|------|
| id | 局處內部 ID（0–8） |
| ou | 局處名稱 |
| probability | Softmax 信心值（10 位小數字串） |

**信心值使用建議**

- ≥ 0.80：高信心，可直接採用
- 0.50 – 0.79：中信心，建議人工複核
- < 0.50：低信心，應由人工判斷

### 3.2 POST `/tfidf` — 關鍵詞萃取

```json
{"preString": "路燈故障，建請建設處處理", "preNum": 5}
```

回傳該文字的 jieba TF-IDF 關鍵詞，輔助理解分類依據。

### 3.3 GET `/health` — 健康檢查

```json
{"status": "ok", "model_loaded": true, "model_type": "BERT (chinese-roberta-wwm-ext)"}
```

## 4. 模型評估解讀

### 4.1 關鍵指標說明

**Accuracy（準確度）** vs **Macro F1**：

| 指標 | 計算方式 | 適用情境 |
|------|---------|---------|
| Accuracy | 答對筆數 / 總筆數 | 快速概覽，但對類別不均衡有誤導性 |
| Macro F1 | 各類別 F1 的算術平均 | 真實評估各局處表現，更公平 |

> 本資料集警察局佔 31%，若模型全猜警察局，Accuracy 就有 31%，但 Macro F1 會接近 0。
> **Macro F1 才是判斷模型品質的主要指標。**

### 4.2 各局處表現（2026-04-20 版本）

| 局處 | TextCNN F1 | BERT F1 | BERT 信心是否穩定 |
|------|-----------|---------|----------------|
| 警察局 | 0.66 | 0.84 | 穩定 |
| 社會處 | 0.70 | 0.86 | 穩定 |
| 教育處 | 0.65 | 0.79 | 穩定 |
| 交通及觀光發展處 | 0.67 | 0.77 | 穩定 |
| 環境保護局 | 0.59 | 0.73 | 穩定 |
| 建設處 | 0.64 | 0.72 | 穩定 |
| 農業處 | 0.51 | 0.71 | 稍弱 |
| 國際發展及計畫處 | 0.53 | 0.65 | 稍弱 |
| **臺東市公所** | **0.28** | **0.55** | **偏弱（recall 42%）** |

### 4.3 臺東市公所偏弱的原因分析

1. **訓練資料偏少**：1,596 筆，僅為警察局的 22%
2. **業務語意重疊**：部分市公所業務（環境、交通）文字特徵與其他局處高度相似
3. **建議處理方式**：增補訓練資料至 3,000 筆以上，或加入 Class Weighting

## 6. 已知限制與優化方向

### 現有限制

| 限制 | 說明 |
|------|------|
| 類別不均衡 | 警察局訓練資料是最少類別的 8 倍，影響少數類別召回率 |
| 臺東市公所 recall 偏低 | 42%，約 6 成此類陳情會被誤判為其他局處 |
| 訓練資料時間範圍 | 資料來自 2021 年，語言與業務用語可能隨時間漂移 |
| 模型無法解釋 | 無法直接說明「為什麼」分到某局處（可用 /tfidf 輔助） |

### 優化方向（依優先序）

1. **Class Weighting**：在 loss 函數中對少數類別加大權重，改善 recall，不需增加資料
2. **增補臺東市公所訓練資料**：目標 3,000 筆
3. **縮短 BERT 重訓週期**：實測 epoch 1 即為最佳，重訓只需約 4 分鐘
4. **信心值閾值機制**：低信心預測自動轉人工審核，而非直接輸出

## 7. 新專案導入指引

> 本 POC 使用的是臺東縣政府舊有案例資料，模型的分類標籤（局處）與訓練語料均屬特定縣市。
> 若要將本系統移植至**新縣市或新業務情境**，需依本章重新設計標籤、清理資料、建立輸入架構。
> **API 介面（/predict、/tfidf、/health）不需修改**，前端可無縫切換。

---

### 7.1 確認輸出標籤（分類目標）

導入新專案的第一步是確認「要分到哪些單位」，即模型的輸出類別。

**需要釐清的問題：**

| 問題 | 說明 |
|------|------|
| 分派對象是哪些單位？ | 列出完整局處清單，例如：建設局、環保局、衛生局… |
| 有沒有「其他」類別？ | 是否有無法分類的案件需獨立一類 |
| 最少需要幾個類別？ | 建議至少 2 類，上限無硬性規定，但類別越多資料需求越大 |
| 業務邊界清不清楚？ | 若兩單位業務高度重疊（如環保局 vs 衛生局），模型容易混淆 |

**產出物：`labels.txt`**

每行一個標籤，順序即為模型 `id`（從 0 開始）：

```
建設局
環境保護局
衛生局
教育局
社會局
警察局
```

> 標籤確定後不應隨意變更順序，否則模型 id 對應會錯亂，需重新訓練。

---

### 7.2 訓練資料格式

本系統的訓練資料格式為**純文字 TSV（Tab 分隔）**，每行一筆：

```
{標籤}\t{陳情文字}
```

範例：

```
建設局	路燈故障已多次通報仍未修復，影響夜間行車安全
環境保護局	工廠排放廢氣，氣味刺鼻，嚴重影響附近居民生活
警察局	檢舉路口違規停車，長期佔用行人穿越道
```

**格式規範：**

| 項目 | 規範 |
|------|------|
| 編碼 | UTF-8（無 BOM） |
| 分隔符 | Tab（`\t`），不是逗號 |
| 每行結構 | 第 1 欄＝標籤、第 2 欄＝文字，無標題行 |
| 標籤名稱 | 需與 `labels.txt` 完全一致 |
| 文字欄 | 不含換行，特殊符號（引號、斜線）無需轉義 |

**資料集切分建議：**

| 資料集 | 比例 | 說明 |
|--------|------|------|
| `cnews.train.txt` | 80% | 訓練用 |
| `cnews.val.txt` | 10% | 訓練過程中驗證，調整超參數 |
| `cnews.test.txt` | 10% | 最終評估，不參與訓練 |

### 7.3 訓練資料清理建議

從業務系統匯出的原始案件資料通常需要清理，以下是常見問題與處理方式：

**常見問題：**

| 問題類型 | 說明 | 處理方式 |
|---------|------|---------|
| 標籤不一致 | 「建設處」vs「建設局」、空白、錯字 | 統一標準化，建立對照表 |
| 文字過短 | 只有 2~3 個字的陳情，語意不足 | 可保留，但建議過濾掉空白或純數字 |
| 文字含個資 | 地址、電話、姓名 | 視需求決定是否去識別化（不影響分類效果） |
| 重複案件 | 同一件事多次陳情 | 可保留（增加樣本），但完全相同的字串建議去重 |
| 類別嚴重不均 | 某單位案件數遠多於其他 | 各類別建議至少 500 筆，差距不超過 10 倍 |
| 標籤模糊案件 | 跨局處業務，人工標注也難決定 | 優先清除，這類資料會讓模型混淆 |

**各類別資料量建議：**

| 類別數量 | 每類最低建議筆數 | 說明 |
|---------|--------------|------|
| 5 類以下 | 300 筆 | 勉強可用，效果有限 |
| 5–10 類 | 500–1,000 筆 | 建議範圍 |
| 10 類以上 | 1,000 筆以上 | 類別越多，每類需要越多資料 |

### 7.4 輸入資料架構設計建議

API 的輸入欄位 `preString` 接收自由文字。實際系統中，一筆陳情可能包含多個欄位（標題、內文、地點等），需決定**送什麼給模型**。

**常見輸入策略：**

| 策略 | 做法 | 適用情境 |
|------|------|---------|
| 僅標題 | `preString = 案件標題` | 標題語意清楚、內文雜訊多 |
| 標題 + 內文前 N 字 | `preString = 標題 + " " + 內文[:200]` | 內文有補充資訊但過長 |
| 完整內文 | `preString = 全文` | 標題不夠、內文品質好 |
| 標題 + 關鍵欄位 | `preString = 標題 + " " + 類型標籤` | 有結構化欄位可輔助 |

> 本 POC 訓練資料使用的是**標題欄**（平均 93 字元，中位數），測試後這個長度對 BERT 已足夠。
> 新專案可先嘗試標題欄，不夠準確再考慮拼接內文。

**BERT 長度限制：**

BERT 最大輸入為 512 tokens（約 400–500 個中文字）。`config.py` 中 `max_length=512`，若輸入更長會自動截斷（保留前段）。若案件內文普遍超長，可改為：
- 截取前 512 字（預設行為）
- 或取標題 + 內文摘要

### 7.5 導入新專案的完整流程（概覽）

```
1. 確認輸出標籤
   └── 整理局處清單 → 建立 labels.txt

2. 準備訓練資料
   └── 從業務系統匯出 → 清理 → 轉為 TSV 格式
   └── 切分 train / val / test（8:1:1）

3. 設計輸入策略
   └── 決定送標題 / 內文 / 組合欄位

4. 訓練模型
   └── 替換 data/ 內的三個 txt 檔
   └── 執行訓練（BERT 約 4–18 分鐘，視資料量）
   └── 確認 val_accuracy 達標

5. 部署驗證
   └── 啟動 API → /health 確認 → /predict 測試數筆
   └── 比對預測結果與人工判斷是否一致

6. 上線前評估
   └── 測試集 Macro F1 > 0.70（建議門檻）
   └── 各類別 recall > 0.50（避免某類幾乎不被預測到）
```

> 步驟 4（訓練）與步驟 5（部署）的詳細操作指引待補，可參考
> `docs/GPU訓練操作手冊_BERT.md` 與本文件第 2 節。
